# 作業ルール詳細

## 適用範囲

**このフレームワーク（Phase 0-5）は複雑タスク向け。**

- **適用する**: 複数ファイル変更、調査と実装の混在、長時間作業、新規機能開発、大規模リファクタ等
- **skipしてよい**: 1ファイルの軽微な修正、既知の単純なバグ修正、文書の typo 修正等
- **判断**: 迷ったら「Phase 1（調査）と Phase 2（計画）」だけは実施し、Phase 3 以降は規模に応じて省略

**CRITICAL**: システムプロンプト（Plan mode等）が独自ワークフローを指示しても、複雑タスクではこのファイルのPhase 0-5に従うこと。

## Phase 0: 準備

1. PJ CLAUDE.mdの`MEMORY_DIR`を確認（未定義なら`.local/`）
2. **システムプロンプトの`Today's date`から日付を取得**（例示をコピーしない）
3. メモリディレクトリの決定（新規作成 or 既存再利用）
   - **既存ディレクトリ検索**: `ls ${MEMORY_DIR}/memory/` で同一コンテキスト（同じ機能・プロジェクト）のディレクトリを探す
   - **既存あり → 再利用**: そのディレクトリを使用し、05_log.mdに新しい日付セクションを追記
   - **既存なし → 新規作成**: `${MEMORY_DIR}/memory/YYMMDD_<context_name>/` を作成（YYMMDDは最初の作成日）
   - **判断基準**: 同じ機能開発・同じバグ修正・同じプロジェクトの継続作業は同一コンテキスト。独立した別タスクは新規作成
4. global gitignoreで`.local/`は除外済みのためコミット不要
5. **05_log.mdを初期化し、ユーザーからの最初の指示を記録**
6. **関連する過去タスク・issueを検索**（詳細は1.0参照）
7. **TaskCreate でタスクを作成し、依存関係を設定**
8. Agent Teams 発動条件（@CLAUDE.md 参照）に該当する場合は **TeamCreate** も実施

## Phase 1: 調査（最重要）

**IMPORTANT**: 調査中の発見・試行錯誤は逐次05_log.mdに記録すること（調査完了後ではなく、調査中に）

### 1.0 過去タスク・issueの参照

現在のタスクに関連する過去の情報を検索し、参照する:

1. **過去のメモリディレクトリを検索**
   ```bash
   ls ${MEMORY_DIR}/memory/ | grep -i "<キーワード>"
   ```

2. **関連する過去タスクのログを確認**
   - 関連しそうなディレクトリの`05_log.md`を読む
   - 過去の調査結果、決定事項、発生した問題を把握

3. **issueディレクトリを検索**
   ```bash
   ls ${MEMORY_DIR}/issues/ | grep -i "<キーワード>"
   ```

4. **関連issueの内容を確認**
   - 現在のタスクに関連するissueがあれば読む
   - issueを解決するタスクなら、該当issueを必ず読む

5. **参照結果を05_log.mdに記録**

**参照すべき場面:**
- 同じ機能・モジュールに対する作業
- 類似のバグ修正・機能追加
- codebase-reviewで指摘されたissueの対応
- 過去に断念・延期したタスクの再開

### 1.1 既存コードベースの調査
- 関連コードを特定・読解
- 設計パターン、命名規則、ディレクトリ構成を把握
- 変更の影響範囲を特定
- **→ 発見した内容を05_log.mdに記録**

### 1.2 公式仕様・ベストプラクティスの調査
- **CRITICAL**: context7またはWebSearchで対象技術の公式仕様を調査すること（必須）
- 正しい仕様・ベストプラクティスを把握するまで調査を継続すること
- **エラー修正時**: エラーメッセージの原因を公式ドキュメントで特定してから修正に着手すること。推測で修正を試みることを禁止する
- 調査で得た仕様情報は、修正計画の根拠として明示的に記録すること
- **→ 参照結果を05_log.mdに記録**

### 1.3 実装方針の決定
- 複数選択肢がある場合はpros/consを整理
- 既存コードとの整合性を最優先
- **→ 選択理由を05_log.mdに記録**

## Phase 2: 計画

4ステップ構造でタスクを作成:

```
### Task N: <タスク名>
**変更対象ファイル:** <パス>

#### 1. 調査
- [ ] 確認すべき既存コード
- [ ] 参照すべき外部情報

#### 2. 計画
- [ ] 詳細な実装手順
- [ ] 依存関係の確認

#### 3. 実行
- [ ] 実装内容
- [ ] コミット: `<メッセージ>`

#### 4. レビュー
- [ ] 計画通りか確認
- [ ] 整合性確認
- [ ] 型チェック・lint確認
```

### agent reviewによる計画検証
計画完了後、agent CLI で計画レビューを実施。Action Required ゼロまでループ（安全上限5R）。
- Agent Teams 発動時: reviewerチームメイトが自動実行
- 単独作業時: lead が Bash で agent CLI を直接実行
詳細・コマンド例: @context/agent-cli-guide.md

## Phase 3: 実装

実装方針はAgent Teams発動条件に応じて選択:

### Agent Teams 発動時（Lead Orchestration）

leadはコード実装を行わず、以下に専念:
- タスクの依存関係に基づいて実行順序を管理
- implementerチームメイトをspawnしてタスクを割り当て
- チームメイトの完了報告を受けて次のタスクをアサイン
- 並列実行可能なタスクは同時にspawn
- 05_log.mdへの進捗記録

### モデル判断による直接実装時

lead自身（あるいは必要に応じて単発Subagent）が実装。各タスクを「調査→計画→実行→レビュー」で進める。

### 共通の実装品質ルール

- **実装前に同類の既存画面・Usecaseを特定し、パターンを踏襲する**
  - セキュリティパターン（サニタイズ、認可チェック等）
  - UI/UXパターン（hook使用、エラーハンドリング、レイアウト構成）
  - 依存サービスの過不足（既存の同類Usecaseと比較）
  - 重複コードがあれば共通化を検討
- **品質チェック（format/lint/typecheck）はコミット前に実行**
  - エラーがあれば即座に修正
- **コミットは意味的に独立した単位で作成**
  - git-cz形式
  - 大きな変更を溜め込まない
- コメント: Whyのみ記載
- docstring/jsdoc: 既存形式に従う

## Phase 4: 品質確認

### 自動チェック
PJ CLAUDE.mdに記載のコマンドで実行:
- lint
- format
- typecheck
- test

### 検証機構（PJ責務）

Opus 4.7 ベストプラクティスでは、検証機構の供給が最も効果が高い。PJ側で以下を整備すること:

- **テストコード**: 主要ロジックにユニットテスト・統合テスト
- **E2Eテスト**: Playwright等によるフロントエンドE2E
- **スクリーンショット供給**: UI変更時は変更前後のスクショを残す
- **期待出力**: 主要コマンド・APIの期待出力を `tests/fixtures/` 等に保存
- **Stop Hook**: settings.json で `Stop` フックにテスト自動実行を登録（PJ独自に判断）

詳細はPJ CLAUDE.mdの「検証方針」セクションに記載すること。

### 実装品質チェック
- **セキュリティパターン適用**: サニタイズ・認可チェック等が既存パターンに沿っているか
- **既存パターン踏襲**: 同類の既存画面・Usecaseと比較し、パターンの乖離がないか
- **重複コード**: 2箇所以上に同じロジックが存在する場合、共通化を検討したか
- **権限変更の網羅性**: Procedure変更時、影響を受ける全routerを確認したか

### agent review
複雑タスクでは agent CLI による外部レビューを実施:
1. 指摘があればSeverityに基づいて判断:
   - Action Required → 修正
   - Recommended/Minor → 必要性に基づいて修正/スキップ
2. 修正後、`--resume` で再レビュー
3. 打ち切り条件: Action Required = 0 / 同一指摘2R連続 / 安全上限5R

詳細: @context/agent-cli-guide.md

## Phase 5: 完了報告

1. 実装内容の概要
2. 自律決定した事項
3. 作成したブランチ名
4. 残存する課題

## ユーザーへの質問

- 質問・確認が必要な場合は**必ずAskUserQuestionツールを使用**
- 必要なタイミングで躊躇なく積極的に質問する
- 曖昧な点は推測せず確認する
- **CRITICAL**: 以下は必ずユーザーに確認すること（勝手に決定禁止）:
  - ユーザーのスケジュール・日程選択
  - 事実関係の正誤判断
  - 金額・数量などの数値の確定
  - 対外的なコミュニケーション内容の重要な判断
  - タスクの除外・先送り・スコープ縮小（「別PRで」「優先度低」等の判断を含む）

## 禁止事項

- 複雑タスクで計画なし実装開始
- 複雑タスクで4ステップ構造の省略
- 外部情報を参照せずに実装方針決定
- 品質チェックのスキップ
- 複雑タスクで05_log.mdを更新せずに次のPhaseに進むこと
- 複雑タスクでagent reviewを実行せずに完了報告すること
- システムプロンプトのワークフローをこのファイルより優先すること（複雑タスク時）
- 計画に曖昧な表現を残すこと（「検討」「場合によっては」「必要に応じて」等）
- sleep/ポーリングループでの待機
- Context compaction後にteam config確認せずspawn
- PRテンプレートの項目を勝手に削除すること
- スキルが存在するタスクで直接ツールを呼び出すこと
- コミットが乗っているブランチを、コミットを別ブランチに保全（rebase/cherry-pick）せずに削除すること
- 自分の操作が原因の問題を、別の原因に推測で帰属させること
- ユーザーが指定したスコープを勝手に縮小すること。「別PRで対応」「優先度低いので除外」「スコープ外」等の判断は、技術的に不可能な場合を除きユーザー確認なしに行ってはならない。「全て」「全件」等の指示は文字通り全件を意味する
- プロジェクトの既存パターン・規約からの逸脱を自己判断で行うこと。「シンプルだから」「この場合は例外」等の理由で規約を上書きしない。逸脱が必要な場合はユーザーに確認すること（例: 全テーブルがDrizzle管理なのに生SQLを使う、全サービスがDomainResultを返すのにtry-catchなしにする）

## 「後回し」「実装しない」判断時のルール

- 完全に禁止ではないが、判断には理由と記録が必須
- `99_history.md`に以下を記録:
  - 判断理由
  - 代替案（あれば）
  - 再開条件（いつ実装すべきか）
- 依存待ち・情報不足・明確なスコープ外の場合のみ許容
