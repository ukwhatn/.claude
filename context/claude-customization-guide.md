# Claude Code カスタマイズガイド（Opus 4.7 対応）

## CLAUDE.md 設計原則

### 200行以下に収める
- **200行以下で92%以上のルール適用率**を達成（研究結果に基づく）
- 400行を超えると約71%に低下
- Claude Codeのシステムプロンプトに既に約50の指示が含まれるため、カスタムルールの実効枠は**100〜150個**
- 詳細は`@context/`への参照で段階的開示（Progressive Disclosure）する

### 記述ルール
- **命令形で書く**:「TypeScript strictモードを使用する」と書き、「このプロジェクトはTypeScriptを使っています」とは書かない
- **ポインタを優先する**: コードスニペットではなく`file:line`の参照を使う（スニペットは陳腐化する）
- **リンターの代替にしない**: コードスタイルの強制はESLint/Prettierにフック経由で委譲する
- **曖昧な表現を避ける**:「適切に処理する」→「エラーはResult型で返し、throwは使わない」
- **強調表現を活用**: `IMPORTANT`, `CRITICAL`, `YOU MUST` で重要な指示を強調

### ドキュメント分離の原則

| 対象 | 配置場所 | 用途 |
|------|---------|------|
| **人間向け** | README.md, docs/ | プロジェクト説明、API仕様、アーキテクチャ図 |
| **エージェント向け** | CLAUDE.md, context/, rules/ | AI向け指示、作業ルール、ファイル形式 |

## ルール管理の2つの仕組み

### @参照（Progressive Disclosure）
CLAUDE.mdから`@context/xxx.md`で参照。参照先は**必要時にのみ**ロードされ、コンテキストを節約。最大5階層まで再帰展開。

### `.claude/rules/` ディレクトリ（v2.0.64+）
各`.md`ファイルが自動的に読み込まれるモジュラールール。`paths`フロントマターでファイルパターンに基づく**条件付きロード**が可能。

```yaml
---
paths:
  - "src/api/**/*.ts"
---
# API開発ルール
- すべてのAPIエンドポイントに入力バリデーションを含めること
```

| 仕組み | ロードタイミング | 用途 |
|--------|-----------------|------|
| `@context/`参照 | 参照元（CLAUDE.md）読み込み時 | 全般的な詳細ルール |
| `rules/`（pathsなし） | 常時 | 常に適用するモジュラールール |
| `rules/`（pathsあり） | マッチするファイルを扱う時のみ | パス固有のルール |

**推奨**: 全般ルールは`@context/`、パス固有ルールは`rules/`（pathsあり）で使い分ける。

## コンテキスト最適化

### .claudeignore（最重要）
`.gitignore`と同じ構文で、Claudeに読ませないファイルを指定。**最大の単一改善**をもたらす。

```
# .claudeignore
package-lock.json    # 30K-80Kトークン節約
node_modules/
.next/               # Next.jsプロジェクトで30-40%削減
dist/
*.min.js
```

### その他の最適化
- **1メッセージ1タスク**: 依存関係の同時保持を防ぐ
- **無関係なタスク間で`/clear`**: コンテキスト混在は精度を約39%劣化させる
- **探索にはサブエージェント**: 中間的なファイル読み取りをメインコンテキストの外に保つ
- **未使用MCPサーバーを無効化**: ツール定義がサイレントにコンテキストを消費

## Skills / Commands / Subagents / CLAUDE.md の使い分け

| 配置先 | 呼び出し | コンテキスト | 用途 |
|--------|---------|-------------|------|
| **CLAUDE.md** | 自動（常時） | メイン会話 | 常時適用ルール |
| **Skills** | 自動判断 or `/skill` | メイン会話 | ドメイン知識、自動トリガー |
| **Commands** | `/command`（**レガシー**） | メイン会話 | ユーザー制御のショートカット |
| **Subagents** | 自動委譲 | **独立** | コンテキスト分離、並列処理 |
| **Agents** | 自動委譲 | **独立** | 専門タスク、ツール制限付き |
| **Agent Teams** | 自動/ユーザー制御 | **独立 + 通信** | 並列・議論・協力 |

### Skillsを選ぶ場合（推奨）
- Claudeが**自動的に使用タイミングを判断**すべき場合
- スクリプトと命令をセットでパッケージ化したい場合
- **クロスプラットフォーム移植性**（Claude Code, Cursor, VS Code等で動作）

### Commands（レガシー）
- `.claude/commands/`に配置するMarkdownファイル
- `$ARGUMENTS`や`$1`/`$2`の位置パラメータをサポート
- **新規作成時はSkills形式を推奨**（Commandsは後方互換のため維持）
- 既存Commandsは正常動作するため、急ぎの移行は不要

### Subagentsを選ぶ場合
- **コンテキストを分離**したい場合（大量の中間結果でメイン会話を汚さない）
- **複数タスクを並列処理**したい場合

**Opus 4.7 の方針**: Subagent の毎回明示的呼び出しは性能低下の原因。明示的に呼ぶのは「複数ファイルへの並列作業」「独立した複数タスク」のみ。それ以外はモデル自身の判断に委ねる。

### Agent Teamsを選ぶ場合
- 5+ファイル変更見込み × 並列実装可能
- 独立タスクが3つ以上ある
- ユーザーが明示的に「チームで」指示
- それ以外は直接実装または単発Subagentを推奨（@context/agent-teams-guide.md 参照）

## Skills 設計のベストプラクティス

### 3つのアーキタイプ
| パターン | 構成 | 用途 |
|---------|------|------|
| **A: Markdownのみ** | SKILL.md | コーディング標準、チェックリスト |
| **B: SKILL.md＋スクリプト** | SKILL.md + scripts/ | バリデーション、ファイル変換 |
| **C: SKILL.md＋MCP/サブエージェント** | SKILL.md + 外部連携 | 外部サービスワークフロー |

**原則: パターンAから始め、必要な場合にのみ複雑さを追加する。**

### descriptionの書き方
「**何をするか**」+「**いつ使うか**」の両方を含める。descriptionはSkill1つあたり約100トークンで常時コンテキストに常駐する。

### 3段階ロードシステム
| レベル | ロードタイミング | トークンコスト |
|--------|----------------|---------------|
| Level 1: メタデータ | 常時（起動時） | 約100トークン/Skill |
| Level 2: SKILL.md本文 | Skillトリガー時 | 5,000トークン未満推奨 |
| Level 3+: references/ | 実行時オンデマンド | 実質無制限 |

### SKILL.md フロントマター

```yaml
---
name: skill-name
description: 何をするか。いつ使うか。
# オプション:
allowed-tools: Read, Grep, Glob        # 使用可能ツールの制限
disable-model-invocation: true          # 手動起動のみ（副作用のある操作向け）
---
```

### Progressive Disclosure
- SKILL.mdは500行以下
- 詳細は`references/`に分割（参照は1階層のみ）
- スクリプト出力のみがコンテキストに入る（ソースコードは入らない）

## ユーザースコープ vs プロジェクトスコープ

| 内容 | 配置場所 | 理由 |
|------|---------|------|
| 作業フロー | ユーザースコープ | 全PJ共通 |
| メモリディレクトリ場所 | **PJスコープ** | モノレポ対応 |
| 品質チェックコマンド | **PJスコープ** | PJ依存 |
| agent cliの使い方 | ユーザースコープ | 全PJ共通 |
| パス固有ルール | **PJスコープ** (`rules/`) | PJ構造依存 |

## ディレクトリ構造（推奨）

### ユーザースコープ

実体はユーザー設定ディレクトリ（`CLAUDE_CONFIG_DIR` 設定時はそのディレクトリ、未設定時は `~/.claude/`）。

```
~/.claude/
├── CLAUDE.md           # グローバル設定（200行以下）
├── context/            # @参照先の詳細ルール
├── skills/             # Skills（推奨）
├── commands/           # Slash Commands（レガシー）
├── rules/              # モジュラールール
└── settings.json       # 権限・環境変数
```

### プロジェクトスコープ
```
<project-root>/
├── CLAUDE.md           # エージェント向け: PJ固有ルール（200行以下）
├── README.md           # 人間向け: プロジェクト説明
├── .claudeignore       # コンテキスト最適化（重要）
├── docs/               # 人間向け: API仕様、アーキテクチャ図
└── .claude/
    ├── settings.json   # PJ共有設定（Git管理）
    ├── context/        # PJ固有の詳細ルール
    └── rules/          # パス固有のモジュラールール
```

## マルチエージェント品質パターン

### Writer/Reviewer パターン（Anthropic公式推奨）
品質重要な作業で、実装セッションとレビューセッションを分離する。実装バイアス（自分が書いたコードを甘く評価する傾向）を排除するため、新しいコンテキストが不可欠。

### Generator/Evaluator パターン（Harness Design）
Anthropic Labsの研究に基づく。エージェントに自身の成果物を評価させると品質が低下するため、**生成と評価を分離**する。
- **Generator**: 機能を実装
- **Evaluator**: 独立した評価者として懐疑的にレビュー
- Opus 4.6ではスプリント分解やコンテキストリセットが不要に簡素化
- タスクの複雑さに応じてEvaluatorの投入を判断

## Opus 4.7 ベストプラクティス

参考: https://qiita.com/ot12/items/06420caf41a34a910c53

### 自律実行を信頼する

Opus 4.7 は自己完結型の実行を前提に設計されている:

- 初回プロンプトに **Goal / Constraints / Acceptance criteria** を整理して渡せば自律実行可能
- ペアプログラミング型の細かい指示・「こまめに」「逐一」等の過度な強制は避ける
- 途中介入を最小化し、長時間セッションは `/focus` `Recaps` で結果中心に確認

### 検証機構の供給（最も効果の高い施策）

検証が無いと自律実行は信用できない。PJ側で以下を整備すること:

- **テストコード**: ユニット・統合テスト
- **E2Eテスト**: Playwright等
- **スクリーンショット**: UI変更時の変更前後
- **期待出力**: 主要コマンド・APIの fixture
- **Stop Hook**: settings.json の `Stop` フックでテスト自動実行（PJ判断）

### Effort Level

- デフォルトは **xhigh**（max は overthinking 傾向）
- 最難関タスクのみ max を選択
- settings.json の `effortLevel` で設定

### 権限プロンプト

- `--dangerously-skip-permissions` の常用は避ける
- Auto Mode（Max plan以上）または `/fewer-permission-prompts` で許可リスト管理
