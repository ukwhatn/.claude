# デバッグ支援技法

systematic-debuggingの各フェーズから参照される3技法。

## 1. Root-Cause Tracing（Phase 1で使用）

エラーが**現れた場所**を直すのではなく、**最初のトリガー**を直す。

- 不正な値・状態を見つけたら、コールスタックを遡って「誰がこの値を作ったか / 誰が不正な引数で呼んだか」を繰り返し問い、発生源に到達するまで遡行する
- 遡行の各段で証拠を取る（ログ・デバッガ・printで実測。推測で層を飛ばさない）
- 発生源で直せば、同じ値が流れる他の経路もすべて直る。エラー出現箇所への対処は同種バグの再発を許す

## 2. Defense in Depth（Phase 4・根本原因の修正後に使用）

根本原因を1箇所直した後、同じ不正データが二度と通れないように、データが通る各層に検証を足す。

| 層 | 役割 |
|---|---|
| Entry Point | 入力の形式・範囲を入口で検証し、不正なら即座に明確なエラー |
| Business Logic | ドメインの不変条件を強制（型・アサーション） |
| Environment Guard | 環境・設定の前提（必須env・接続先）を起動時に検証 |
| Debug Instrumentation | 将来の調査のために境界のログを残す |

単一の検証は「バグを直した」、多層の検証は「バグを不可能にした」。ただし全層に機械的に足すのではなく、実際にバグが通った経路に足す。

## 3. Condition-Based Waiting（テスト・非同期処理の不安定さに使用）

任意の固定時間待ち（sleep / setTimeout N秒）は、遅い環境で壊れ速い環境で時間を浪費する。**時間ではなく条件を待つ**。

- 「Nms待つ」を「条件Xが真になるまでポーリング（上限付き）」に置き換える
- テストフレームワーク・ライブラリの待機プリミティブ（waitFor / Eventually 等）を優先する
- 上限到達時は「何の条件が満たされなかったか」が分かるメッセージで失敗させる
