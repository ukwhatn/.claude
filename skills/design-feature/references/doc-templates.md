## 01 実装マスタ (`01_requirements_skeleton.md`)

### 目的

実装担当が**そのまま設計に落とせる**粒度で What + How を書く。

- 関数・クラス・ファイル名・行番号を明示してよい
- DB スキーマ・enum 追加箇所を具体的に書く
- 既存パターンの参照を `path:Lnum` で示す
- agent review を通してから「実装マスタ」として確定する

### 書き方の原則

- **設計理由を本文に書く**: 「設計理由: …」を一文添える（02 では削る）
- **実コード参照は `path:Lnum`**: 既存パターンを引用するとき必須
- **複数案ある時は AskUserQuestion で確定**: 不採用案は `30_decisions.md` に逃がす
- **「別途決定」を残さない**: 確定するか「やらないこと」に移す
- **テーブル / コード断片は最小限のサンプルで OK**: 全カラム列挙ではなく追加・変更箇所のみ

## 02 システム要件書 (`02_system_requirements.md`)

### 目的

Biz / PdM / RP / CS が**実装を意識せず**仕様を理解できる粒度で What のみ書く。

- 既存の Confluence「[InstAcc] Ph.1 システム要件」スタイルに揃える
- 関数・クラス名・ファイル名・行番号は**書かない**
- Why は注記程度（「※」「→」1-2 行）
- 取り消し線でスコープ外を表現可

### 標準章構成

```markdown
# [<プロジェクト略号>] <フェーズ名> システム要件

| 項目 | 内容 |
|------|------|
| 親エピック | <JIRA キー> |
| 対象スコープ | <1-2 文で要約> |
| 実装マスタ | `<ディレクトリ>/01_requirements_skeleton.md` |
| Figma 抽出文言 | `<ディレクトリ>/12_figma_text_extract.md` |
| 既存実装事実 | `<ディレクトリ>/13_xxx_code_facts.md` |
| 設計判断記録 | `<ディレクトリ>/30_decisions.md` |

---
```

### 書き方の原則

- **箇条書き多用**: 段落は最小限、原則箇条書き
- **サブセクション X.Y / X.Y.Z**
- **表は対比用途のみ**: 「A の場合 / B の場合」「親作の子ども / T2 経由の子ども」
- **常体・宣言形**: 「〜する」「〜できる」「〜のみ」
- **AI 翻訳調禁止**: 「〜することが可能」「実施する」「対応する」「活用する」を削る
- **What → 仕様レベル可**: API レスポンスのフィールド名・scope 名・claim 名・エラーコード・画面 URL 例
- **取り消し線**: 「~~5. 昇格~~ → Phase1スコープ外」のように使う
