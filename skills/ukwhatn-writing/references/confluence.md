# Confluence / ドキュメントの書き分け

設計書・手順書・告知・相談・メモなど。共通する核は **トップダウン（目的・背景を先に）＋必ず「なぜ」を添える＋用途別の固定テンプレ**。読み手で常体/敬体を切り替える。

---

## 文体の基本

- **構成はトップダウン**: 「概要/背景 → 目的 → 本論 →（検討事項/確認事項）」。
- **冒頭にメタ説明**: 「このページ/この章は〜のためのものです」とページの目的・使い方を最初に書く。
- **常体/敬体は読み手で切替**:
  - 自分・チーム内の思考整理（設計書・要件・負債メモ・リサーチ）→ **常体**（だ・である・体言止め）
  - 対外告知・他チーム/Bizへの相談・手順の指示文 → **敬体**（です・ます。指示は「〜してください」）
  - 同一ページ内でも、地の文は常体・読み手への注意書きは敬体、と混在しうる
- **語彙**: API名・テーブル名・scope名は原語のまま `code`。概要・背景は平易語で噛み砕く。用語・状態は「」で定義・強調。
- **一文**: 地の文・箇条書きは短く体言止め＋子項目で理由を1行補足。背景説明だけは因果を省略せず1文に詰める。

---

## 用途別テンプレ

### 設計書 / 思考整理（常体）
`# 概要` → `**背景:**`（番号付き）→ `**目的:**` → 本論（要件/画面/サーバー/DB/テスト等）→ **`# 検討事項・設計判断`**。
末尾の設計判断は **代替案 → 採用理由をセット**で残す:
```
## <判断したテーマ>
<採った方針を1文>

**代替案として検討されたもの:**
* <案A> → <却下理由>
* <案B> → <却下理由>

**採用理由:**
* <理由>
```
例外・未実装は `> **Note:** …` で補足。

### 運用 / 手順書（敬体・指示）
固定テンプレ: **実施日 / 参加者 / 承認 / 背景・経緯 / 対象 / 手順（事前確認・本番作業・事後確認）**。
- 非自明な手順には **「〜のため」と理由を併記**（例: 「意図しない重複実行を防止するため」）。自明な手順には付けない
- コマンドはコピペで動く完全形＋コメント入り
- 事後の安全確認はチェックボックス（`- [x] secret, 個人情報をクリップボードから破棄した`）
- secret/個人情報の警告は冒頭固定
- 選択肢が複数あるときは `<pros>/<cons>` で比較し **「ukとしては〜を推したいです」** と推奨を主語付きで明示

### 対外告知（敬体）
冒頭に引用パネルで目的提示:
```
> このページは、<対象>に対して<目的>のためのものです。
```
番号付き見出し（`## 1.`）。重要な注意は**太字**、不確実性は `※ 現時点での予定です` 等のコードスパンで前置き。読み手の作業負担（例: parser改修要否）に踏み込んで「対応要否を含めご検討をお願いいたします」と配慮。

### 相談 / 意思決定依頼（敬体）
「現状・問題点」→ `## 確認事項`（`### ①` `### ②` と番号）。
- 各確認に **技術的トレードオフを添えて判断材料を渡す**（「実装は可能ですが、〜リスクがあります」）
- 質問は相手に選択肢を残す形（「〜でしょうか」「〜の余地はありますか」）
- 柔らかさは「ご相談させていただきたいです」で出す。絵文字（🙏🙇‍♂️）は依頼・相談の締めだけに1個、過剰に使わない

### 負債 / メモ（常体）
冒頭1文で目的定義 → 箇条書きの体言止め・短文に徹する。子項目で理由を簡潔に（「検証時不便」）。未決は疑問形で残す（「分けた方が良い？」）。

### リサーチ / 概念整理（常体）
抽象概念は **「定義 / 具体例 / 〜とのつながり」の3点セット**で毎回同じ型に。As-Is/To-Be表・なぜなぜ分析・ペルソナ等のフレームを使う。括弧で「例：〜」と具体に落とす。

---

## 構造化の手段
- **表**: 比較・対応関係（種別×差分、As-Is/To-Be、API×制限）。最も得意な整理手段
- **コードブロック**: コピペで動く完全形＋前提コメント
- **疑似図**: 画面遷移・階層をインデント＋矢印で表現
- **チェックボックス**: 事前/事後確認、進捗管理

## 読み手への配慮（誠実さの癖）
- 非自明な手順・判断・リスクには「なぜ」とトレードオフを添える（自明な手順には付けない＝冗長回避）
- 確定/未確定を区別（`※` `〜疑惑` `〜と思っている`）
- 自分の意見は `uk所感)` `ukとしては` とラベルで分離する
