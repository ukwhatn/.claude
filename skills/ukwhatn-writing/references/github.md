# GitHub の書き分け

PR概要欄 / レビュー指摘 / レビュー返信 の3つ。共通して **絵文字・感嘆符は控えめ（指摘時はほぼ無し）**、技術用語は原語＋ `inline code`、件数・パス・ハッシュは具体で書く。

---

## 1. PR概要欄

### 固定見出しテンプレ（社内チーム）
日本語見出しを順に並べる。**`## やらなかったこと` を立てる**のが最大の特徴（スコープ外があれば理由付きで明示、なければ「特になし」と1行で書く。軽微PRでも見出しは残すが中身は簡潔に）。

```
# 概要
<1〜2文。何をするPRか。背景がいるなら一言>
- closes #<Jira/Issue番号>

## 動機
<なぜ必要か。背景・経緯。関連スレッドのリンクを添える>

## やったこと
- `path/to/file.ts` に `xxx` を追加。<何をするか>
- <体言止め・常体で箇条書き。inline codeでパス/関数/設定値を囲む>

## やらなかったこと
- <スコープ外を明示。理由を「— …のため」「（…のため）」で添える>

## 影響範囲
- <影響する機能・経路>

## テスト方法
ローカル:
- [x] `bun run typecheck` 通過
ユーザー手動確認:
- [ ] <手動確認項目>

## レビュー時チェック
- <見てほしい観点: 境界ケース、すり抜け経路など>
```

### バリエーション
- **軽微PR**: `## やったこと` を数行、`## やらなかったこと` は「特になし」、`## レビュー時チェック` は **「特にありません（普通にレビューしてください）」**。
- **大規模PR**: カテゴリ別見出し＋件数併記（`### セキュリティ修正（8件）`）、冒頭に総数集計（`全73件（critical 4, major 22, minor 34, trivial 13）`）。`| カテゴリ | 件数 | 対応 |` の表で整理。
- **OSS/外部repo**: 英語見出し（`## Summary` / `## Test plan`）＋箇条書きのみの簡素版に切替。
- **運用注意・PoC警告**: 冒頭に GitHub alert。
  ```
  > [!IMPORTANT]
  > developマージ後、terraform applyを各環境に対して実行します
  ```
- **同時マージ等の運用指示**: 末尾に `---` 区切りで太字注意。`**本PRは hotfix の同時PR3本の1つです。…可能な限り同時にマージしてください。**`
- `## やったこと` の中に `### 原因` を入れ子にして根本原因を説明する型もある。

### 実例（抜粋・原文ママ）
`## 検証` は `## テスト方法` の別名（検証手順を書くPRで使う短縮版）。
```
## 概要
dependabot PR #288（dev-dependencies group in /batch）で `ty` が `>=0.0.40` に更新されると、厳格化した型チェックで既存コードの型エラーが顕在化し `Lint (Python)` が失敗していた。型注釈の追加のみで解消する（ロジック変更なし）。

## 検証
- `uv run ty check`（ty 0.0.42 / 0.0.40相当）→ error 0件（残る not-iterable 警告3件は pyproject で `warn` 設定済みのためexit 0）
```

## 2. レビュー指摘（レビュアーとして）

**根拠の一次ソースURL ＋ 代替案** をセットにする。**事実は断定し、提案・推測だけぼかす**。絵文字なし。
- **断定してよい**: 仕様違反・CI失敗・再現済みバグなど事実（「`個人情報を含ませないでください。` に違反します」「lintも落ちています」）
- **ぼかす**: 確証のない推測・あるべき実装の提案（「〜ではないかと思います（おそらく）」「氏名はDBから引くべきかと思います」）

### 型
- 推測指摘: 「〜ではないかと思います（おそらく）」＋公式ドキュメントURL
- 代替案併走: 指摘だけで終わらず「`team_reviewers` を使うことでここの処理自体を削除できないでしょうか」
- 規約・仕様違反: 端的に事実指摘「`個人情報を含ませないでください。` に違反します。氏名はDBから引くべきかと思います」（違反は断定、改善提案はぼかす）
- 判断を委ねる疑問形: 「このガードは…には不要の判断でしょうか。」
- コード片・ログは ```code block``` で引用してから論じる

### 実例（原文ママ）
- `[lint] も落ちていますが、permissionsにmembers scopeは存在しないかと思います`（直後にドキュメントURLのみ連投）
- `このAPI叩くのに org:read が必要なので、GHAのpull_request targetだとPATを別途発行しない限り叩けないのではないかと思います（おそらく）`
- `timezoneいれておきたいです`（＋ドキュメントURL）

## 3. レビュー返信 / 作業報告（自分のPRで）

受領は **「ありがとうございます！」＋次アクション宣言**。指摘時よりやや柔らかく、感嘆符が出る。

### 実例（原文ママ）
- `ありがとうございます！念の為update branchしてからマージします`
- close/保留判断（短く逃げ道付き）: `一度closeします、必要であればreopen / recreateしてください`

### 大きめの作業報告は構造化
見出し＋番号＋表。`**問題**:` → `**対応**:` → `**TODO**:` のラベル太字で因果を整理。
```
## dependabot一括更新に対する手動修正
以下の修正を手動で追加しました。
### 2. tsc型エラー修正 (`37a49a231`)
71パッケージ更新に伴い39件のtsc型エラーが発生。以下を修正:
| カテゴリ | 件数 | 対応 |
|---------|------|------|
| react-hook-form 判別共用体変更 | 35 | ... |
### 3. `@vonage/server-sdk` を 3.19.4 に固定
**問題**: ESM-onlyパッケージを `require()` できず77テストスイートが失敗。…のため対応困難。
**対応**: `@vonage/server-sdk` を `3.19.4` に固定。
**TODO**: 別PRで Jest ESM対応と合わせて個別対応が必要です。
```

## markdown の癖
- インラインコードを多用（パス・関数・変数・設定値・コマンドは必ず `` ` `` で囲む）
- チェックリストは担当主体で分ける（`ローカル: [x]` / `ユーザー手動確認: [ ]`）
- 対応関係は表、Slack依頼文/bot指摘/引用は `>` blockquote、運用注意は `> [!IMPORTANT]`
- リンクは `[ラベル](URL)`。公式ドキュメント・Slackスレッド・Jira・関連PR番号(`#3984`)を密に相互リンク
- 強調 `**…**` は要点語にピンポイント（乱用しない）
- 「やったこと」箇条書きは体言止め/常体（「〜を追加」「〜化」）、地の文・対人コメントは敬体
